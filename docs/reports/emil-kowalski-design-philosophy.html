Emil Kowalski 技能体系的设计哲学剖析

当 AI 可以写出任何代码，知道"什么是对的"比知道"怎么写"更重要

一位设计工程师把多年的品味训练编码为五套 Claude Code 技能，覆盖从动画审计到 Apple 流体界面的 Web 翻译。这不只是一组开发工具，也是一套在 AI 时代保持设计判断力的方法。

核心判断 Emil Kowalski 的技能体系将隐性设计品味外化为精确的可执行规则，在 AI 生成代码的时代，把"判断什么是对的"置于"生成更多代码"之上。这套体系真正的力量不在于告诉 AI 该加什么动画，而在于告诉 AI 该删什么。

关键发现

01品味可训练：将主观的"感觉不对"转化为精确的 cubic-bezier 数值和可追溯的推理链
02频率优先：动画的存在权由其出现频率决定，键盘快捷键上的动画不是"可以优化"，是"必须删除"
03物理正确性：从 damping ratio 到 scale(0.97)，整个体系将现实世界的运动规律编码为 CSS 参数
04知识金字塔：哲学→原则→规则→参数→文件路径，每一层推导下一层，使体系可操作
01 / 品味训练

品味是可训练的

Emil Kowalski 整个技能体系建立在一个前提之上：品味不是个人偏好，而是训练出来的本能。这被落实为具体的操作姿势、审查标准和精确数值。

五个技能反复出现的立场是：默认标记问题，认可是挣来的（"approval is earned, not assumed"）。review-animations 将审核者角色定义为"a senior design engineer with a brutal eye for craft"，一个标准极高的高级设计工程师。这一立场把看似主观的审美判断转化为可辩护的专业标准。

这套体系将"感觉不对"这种模糊的直觉，转化为可追溯的推理链。以弹窗动画为例：

用户说"这个弹窗出现的方式有点奇怪"→ 检查发现 transform-origin: center → 对照规则：弹窗应从触发器缩放，而非视图中心 → 修复为 transform-origin: var(--radix-popover-content-transform-origin) → 用户不再觉得奇怪，但说不清为什么。

这条链上的每一步都可验证。规则不是"更好的动画"，而是精确的 CSS 属性值。Emil 甚至将 Apple 的物理参数，damping ratio、response time，直接映射为 Web spring 库的具体配置：{ type: "spring", duration: 0.5, bounce: 0.2 }。从 WWDC 演讲到 Motion 库的 API 调用，中间没有猜测。

在足够精确的规则体系下，"好品味"是可操作的。它不是少数天才设计师的专属天赋，任何遵循规则的人都能接近它。这也解释了为什么这些技能是为 AI agent 设计的：agent 没有天生的品味，但可以完美地执行精确的规则。

技能定位

review-animations 的开场白："Your bias is toward motion that feels right, not motion that merely runs. A transition that 'works' but feels sluggish is a regression, not a pass."

将审美判断操作化，这是 Emil 体系与一般设计指南最本质的区别。大多数设计指南告诉你"要有好的动画"，Emil 告诉你"用 cubic-bezier(0.23, 1, 0.32, 1)，200ms，从 var(--radix-popover-content-transform-origin) 开始"。

02 / 频率优先

动画不是装饰，由出现频率决定做不做

在整个技能体系中，最核心的决策框架不看"这个动画好不好看"，看的是"用户一天会看到它多少次"。它把动画从视觉装饰的讨论拉回到功能层面。

Emil 建立了一个严格的四级频率-动画对应表，同时出现在 emil-design-eng、review-animations 和 improve-animations 三个技能中：

使用频率典型场景动画决策理由
100+ 次/天键盘快捷键、命令面板零动画。永远不。重复数百次的动画让操作感觉迟钝和脱节
几十次/天hover 效果、列表导航删除或大幅缩减频繁的装饰性动效累积为认知负担
偶尔模态框、抽屉、toast标准动画值得用运动来解释空间关系和状态变化
罕见/首次onboarding、庆祝、反馈可以加愉悦感低频场景允许更多装饰性和情感化设计

这个框架反直觉的地方在于：它把"删除动画"作为最高优先级的修复手段。在 review-animations 的修复层级（Remedial Preference Hierarchy）中，排在第一位的不是"修复缓动"、不是"优化性能"，而是"删除动画"。Emil 以 Raycast 为典范，一个每天使用数百次的应用，它的命令面板完全没有打开/关闭动画，而这被视为"最优体验"。

这种立场是对当前 AI 生成 UI 趋势的一种批评。AI 模型倾向于过度动画化，每个元素都要 fade in、slide in、scale up，仿佛动画本身就是质量的标志。Emil 的框架反过来：动画是成本，消耗用户的注意力和时间，只有在它能解释空间关系、提供反馈或防止突兀变化时，这笔成本才值得付出。

如果动画的唯一目的是"看起来酷"，而用户每天要看到它几十次，那就不做。

修复层级

review-animations 的 Remedial Preference Hierarchy 中，9 个修复等级的排序：删除动画 → 缩减动画 → 修复缓动 → 修复原点 → 改为可中断 → 移至 GPU → 非对称时间 → 打磨 → 无障碍与一致性。

"Never animate keyboard-initiated actions" 是技能体系中最不容妥协的规则之一。它不是一个建议，它出现在"Non-Negotiable Standards"和"Hard Rules"中，措辞是"Never. Ever."

03 / 物理正确性

动画必须遵循现实世界的运动规律

Emil 的体系设定了动画的物理正确性标准。这不是隐喻，它意味着动画中的每一个元素都应该像现实世界中存在物理实体一样运动。

最基础的规则也最不留余地：永远不要从 scale(0) 开始动画。理由是：

Nothing in the real world appears from nothing. Nothing in the real world disappears and reappears completely.

现实世界中没有任何东西从虚无中出现，也没有任何东西完全消失又重现。因此，进入动画应该从 scale(0.95) + opacity: 0 开始，即使是最微小的初始尺度也让元素有一个"存在但尚未可见"的状态。以下是体系中的核心物理规则：

空间锚定

弹窗和下拉菜单必须从触发它们的按钮缩放出来，不是从视口中心。这保留了操作与结果之间的空间关系，让用户下意识地理解"这个面板来自我点击的地方"。CSS 实现是精确的：

/* Radix UI */
.popover { transform-origin: var(--radix-popover-content-transform-origin); }

/* Base UI */
.popover { transform-origin: var(--transform-origin); }

/* 例外：模态框保持居中，它们不锚定到特定触发器 */
.modal { transform-origin: center; }

按压反馈

每个可按压元素必须在按压瞬间给予反馈，transform: scale(0.97) 配合 transition: transform 160ms ease-out。缩放幅度在 0.95-0.98 之间，模拟物理按压的微妙形变。

弹簧而非关键帧

任何手势驱动的、可中断的运动都应使用弹簧动画。CSS @keyframes 从中断处重新从零开始，而弹簧从当前值继续，携带速度和方向。这在物理上是正确的：现实世界的运动不会因为你的手改变了主意就从起点重放。

非对称时间

人做决定需要时间（按压确认 2s），系统的响应应该迅速（释放 200ms）。这个原则被编码为具体的 CSS 模式：

/* 释放：快速响应 */
.overlay { transition: clip-path 200ms ease-out; }

/* 按压：缓慢而从容 */
.button:active .overlay { transition: clip-path 2s linear; }

用户不需要学习动画语言，因为他们已经生活在物理世界中。符合物理直觉的运动不需要解释，它感觉"对"，用户说不清为什么。当动画违反物理直觉（从 scale(0) 开始、从中心而非触发点弹出、按压没有即时反馈），用户会感到一种说不清的"不对劲"。

来自 AUDIT.md

"Never scale(0) — nothing in the real world appears from nothing. Target: scale(0.9–0.97) + opacity: 0."

物理正确性不是隐喻式的设计语言，它是可验证的。在慢动作回放下，一个从 scale(0.95) 开始的弹窗和从 scale(0) 开始的弹窗之间的差异，在逐帧检查中清晰可见。

04 / AI 时代

专业知识不是被 AI 取代，而是被 AI 放大

Emil 在 README 中写下了整个项目最关键的一句话："AI doesn't replace domain expertise, it amplifies what you can get out of it."

他观察到，当前的 AI agent 在生成动画代码时会反复犯低级错误，为进入动画选择 ease-in（应该用 ease-out），使用实线边框替代半透明阴影，从 scale(0) 开始弹窗动画。这些错误单独看微不足道，但它们会"compound and make your interface either amazing, or just... not that great"，累积起来，要么成就卓越，要么沦为平庸。

Emil 的做法是专家知识固化：将设计工程专家的判断编码为精确的规则，让任何 agent，包括能力较弱的廉价模型，都能执行出符合高标准的结果。这个思路在 improve-animations 技能的工作流中看得最清楚：

Phase 1Recon 侦察映射技术栈、运动位置、组件库、频率分布
Phase 2Audit 审计八大类别并行审查，子 agent 按类别或区域分工
Phase 3Vet 确认主 agent 逐条复核代码引用，剔除误报、重复和设计意图
Phase 4Plan 执行自包含计划，精确到 cubic-bezier、文件路径、验证步骤

在设计工程中，判断力是稀缺资源。昂贵的模型（如 Opus）应该用于需要设计判断力的阶段：哪些动画值得修、怎么修、优先级如何。然后将判断结果编译为精确的执行规格，交给任何 agent，包括廉价模型，去逐行实施。

"自包含计划"（self-contained plans）是这个策略的核心做法。一份计划不是"优化下拉菜单的缓动"，它包含：

  • 当前代码的确切片段和文件位置（src/components/dropdown.css:14）
  • 目标状态的确切 CSS 值（cubic-bezier(0.23, 1, 0.32, 1)，不是"更好的缓动"）
  • 仓库已有的约定和典范文件位置（"缓动 token 放在 src/styles/tokens.css"）
  • 逐步骤的编辑指令和硬性边界
  • 验证步骤，包括机械检查（typecheck）和感觉检查（慢动作回放、逐帧检查）

这样的计划可以被一个"zero context and zero taste"的 agent 执行。

来自 SKILL.md

"Write for the weakest executor: exact file paths and current-code excerpts, the exact target values... pulled from AUDIT.md, never approximated."

这与"prompt engineering"有本质区别。Prompt engineering 试图通过自然语言指令引导模型行为；Emil 的策略是将判断从执行中彻底分离，让执行者不需要判断。

05 / 复利效应

不可见细节的复利："一千个几乎听不见的声音和谐地合唱"

Emil 引用了 Paul Graham 的一个隐喻来描述设计工程的本质：一千个几乎听不见的声音和谐地合唱。没有一个细节单独被用户注意到，但它们的总和定义了产品给人的感觉。

这个隐喻点出了 Emil 体系的逻辑：每个单项规则看起来都微不足道，但聚合效应是质变的。弹窗从触发点而非中心缩放，用户不会注意到，就像你不会注意到合唱团中某个人唱对了音。但如果十个弹窗都从中心缩放，用户会感到一种说不清的"不对劲"。

Emil 在设计 Sonner（一个拥有 1300 万+周下载量的 toast 库）时积累的原则，把这种思路体现得最清楚：

处理边缘情况于无形

Sonner 在浏览器标签页隐藏时暂停 toast 计时器。用伪元素填充堆叠 toast 之间的间隙以保持 hover 状态。在拖拽开始时捕获指针事件。这些细节用户永远不会主动注意到，"Users never notice these, and that is exactly right."注意到它们意味着出了问题。

好的默认值胜过一切可选项

"Good defaults matter more than options."大多数开发者不会自定义 toast 的缓动曲线或时间参数。他们只是调用 toast()，期待它看起来不错。因此，默认值必须已经代表最高标准，不是"够用"，而是"优秀到不需要修改"。这解释了为什么 Emil 的体系中遍布精确的默认值：按钮反馈 160ms、弹窗 200ms、缓动曲线 cubic-bezier(0.23, 1, 0.32, 1)。这些不是"推荐值"。它们是"如果你不主动选择，这就是你得到的，所以它必须是对的"。

一致性本身就是一种功能

Sonner 的动画比典型 UI 动画略慢，使用 ease 而非 ease-out，以营造"优雅"而非"快速"的感觉。这个选择不是孤立的，它与 toast 的视觉设计、文档站点的风格、甚至库的名称（"Sonner"在法语中意为"响铃"）保持一致。Emil 称这为 "cohesion matters"：当缓动、时长、视觉设计和品牌名称全部指向同一种感觉时，产品给人的印象是不可分割的。

All those unseen details combine to produce something that's just stunning, like a thousand barely audible voices all singing in tune.
Paul Graham

Sonner 原则

6 条原则来自 Emil 构建 Sonner 的经验，被编码在 emil-design-eng 技能中：DX 至上、好的默认值、命名创造身份、无形处理边缘情况、transitions 而非 keyframes、优秀的文档站点。

"Cohesion matters" 是最容易被忽视的原则。它解释了一个反直觉的现象：为什么两个功能相同的组件，一个"感觉好"，另一个"感觉一般"，因为前者在缓动、时长、视觉、命名上是一致的，后者是拼凑的。

06 / Apple 传承

将 Apple 的流体界面哲学翻译到 Web 平台

apple-design 是五个技能中最长、最详尽的一个，17 个章节，从响应延迟到排版光学尺寸。它做的事情是系统性的：将 Apple 在 WWDC 设计演讲中阐述的流体界面哲学，逐条翻译为 Web 平台的可执行代码。

Apple 的哲学建立在原生平台的底层机制上：CADisplayLink、UIKit 的弹簧原语、Core Animation 的 additive animations。Emil 的工作是找到 Web 平台的对应物：

Apple 概念原生实现Web 翻译精确值
阻尼比 (damping ratio)UIKit springMotion bounce 参数damping: 1.0（默认无弹跳）
damping: ~0.8（动量交互）
响应时间 (response)UIKit springMotion duration 参数0.3–0.4s（UI 默认）
动量投射UIScrollView decelerationJS projection functionproject(v, d=0.998) = (v/1000)×d/(1−d)
速度交接additive animationsspring initial velocitygestureVelocity / (target − current)
显示同步时钟CADisplayLinkrequestAnimationFrame每帧回调
毛玻璃材质UIVisualEffectViewbackdrop-filter: blur()blur(20px) saturate(180%)

翻译中最关键的，是可中断性的处理。Apple 的核心原则是"界面感觉有生命，当运动从当前屏幕值开始、继承用户速度、投射动量、并可在任何瞬间被抓住和反转"。在原生平台上，这通过 additive animations 实现，你可以随时向正在运行的动画添加新的动画，系统会从当前值开始混合。在 Web 上，Emil 的解决方案是：

  • 用弹簧而非关键帧处理手势驱动运动，弹簧从当前值开始，携带速度
  • 用 CSS transitions 而非 @keyframes 处理动态 UI，transitions 可以从中断点重定向
  • 将 2D 运动分解为独立的 X 和 Y 弹簧，避免 X 和 Y 速度不同时单个弹簧失同步
  • 在反转时混合速度而非硬切，选择能在重新定向时携带速度的 spring 库

它的目标是在 Web 约束下重建 Apple 物理模型的等效行为。用户感觉不到底层平台的差异，因为运动的物理特征被完整地保留了下来。

核心引用

Apple 的原始表述："When we align the interface to the way we think and move, something magical happens — it stops feeling like a computer and starts feeling like a seamless extension of us."

这 17 个章节的精妙之处在于：它不是 Apple HIG 的摘要翻译，而是从物理模型出发，在每个概念上找到 Web 平台的等效表达。最终的产物不是"Apple 风格的设计指南"，而是"Apple 物理模型的 Web 实现手册"。

07 / 克制优先

克制才是真正的设计功力

Emil 技能体系最核心的主张不是"加什么"，而是"删什么"。在一个以"更多功能、更多动效、更多交互"为默认追求的设计文化中，这是一个少见的立场。

键盘快捷键上的动画不需要优化，它需要被消灭。"Never. Ever."的措辞没有留任何余地。装饰性动画在每天高频出现的元素上是设计债务，不是风格选择。当不确定时，删除它。

这种克制的设计姿态体现在三个层面：

对 AI 生成 UI 的批判

AI 模型偏爱动画，它们倾向于为每个元素添加淡入、滑入、缩放，仿佛动画多多益善。Emil 的体系是对这种倾向的系统性回应。技能的自述中有一句定位很准："This is your shortcut to great interfaces. A shortcut to stand out in a sea of slop."这里的"slop"（粗制滥造之物）指的就是当前 AI 生成 UI 中普遍存在的过度动画化倾向。在一个人人都能用 AI 生成界面的世界里，知道什么不该做比知道怎么做更稀缺。

对设计文化的反叛

主流设计文化奖励"可见的工作"，一个精心设计的微交互比一个被删除的动画更能展示设计师的存在感。Emil 的体系反过来：最好的设计工作是用户看不见的。这不是美学选择，而是一种专业伦理：把用户的体验质量置于设计师的自我表达之上。

作为时间尊重的设计

动画消耗用户的时间。一个 300ms 的动画，每天触发 100 次，一年就是 3 小时的等待。Emil 的体系在操作层面落实了这一点：高频操作零动画，低频操作才允许装饰性表现。以用户时间为货币的设计经济学。

隐喻

Emil 将技能定位为在"sea of slop"（粗制滥造之海）中脱颖而出的捷径。这个措辞直接呼应了他对当前 AI 生成 UI 质量状况的判断。

克制作为一种设计姿态，在技能体系的修复层级中被制度化了：修复的第 1 优先级是"删除动画"，第 2 优先级是"缩减动画"，第 3 优先级才是"修复缓动"。删除优先于优化，这是一个很少有人敢明确陈述的立场。

08 / 知识架构

从哲学到文件路径：知识金字塔的完整推导链

Emil 的技能体系构建了一个完整的知识金字塔，从抽象的哲学信念到具体的文件路径，每一层都从前一层推导出来。这个架构是该体系能同时保持深度和可操作性的原因。

五层知识金字塔

每一层都是上一层的推导，形成了一个从信念到执行的完整链条。任何一层断裂，体系都会失去可操作性。

Layer 1 哲学层，为什么做

品味可训练、不可见细节产生复利、克制才是真正的设计功力、专业知识不可被 AI 取代

Layer 2 原则层，做什么

频率驱动决策、物理正确性、空间一致性、可中断性、非对称时间、认可非默认

Layer 3 规则层，怎么做

进入/退出用 ease-out、UI 动画 <300ms、从 scale(0.9-0.97) 开始、弹窗从触发器缩放

Layer 4 参数层，具体值

cubic-bezier(0.23, 1, 0.32, 1)、bounce: 0.2、duration: 0.5、按钮反馈 160ms、弹窗 150-250ms

Layer 5 文件位置层，写在哪里

plans/NNN-slug.md、src/styles/tokens.css、var(--radix-popover-content-transform-origin)

这个金字塔最关键的属性是单向推导性：哲学指导原则，原则推导规则，规则指定参数，参数落到文件位置。你可以从"物理正确性"（Layer 2）一路推导出"按钮应该用 transform: scale(0.97) 配合 160ms ease-out 写在 .button:active 中"（Layer 3→4→5）。反过来也成立：如果一个开发者在代码审查中看到 transition: all 300ms ease-in，他可以一路追溯到被违反的原则（物理正确性）和哲学（品味不可妥协）。

这个架构也解释了为什么 Emil 选择"技能"（skills）而非传统文档作为媒介。技能在 Claude Code 中每次被调用时都会被加载为 agent 的操作系统指令：持续激活的行为规范，不是"阅读后即忘"的参考资料。这是一种新的知识传递形态：它编程 agent 的行为，而不是教人。

在一个 AI 可以生成任何代码的世界里，知道"什么是对的"比知道"怎么写"更重要。Emil 的技能体系用 5 个文件、数千行精确规则，回答了那个更困难的问题。

这就是"固化品味"的机制。传统上，设计品味通过师徒关系传递，需要数年的共事、观察和反馈。Emil 把这些隐性知识逐条提取出来，编码为"agent 可能会犯的所有小错误以及如何修复它们"，变成了一套可验证、可执行、可继承的设计判断系统。

技能作为媒介

技能（skills）不同于传统文档的关键：它们在 agent 每次调用时被加载为操作系统的指令，不是"建议"或"参考"，是 agent 行为的一部分。知识传递从"教人"变成了"编程 agent"。

这个知识金字塔的每一层都可以独立验证。你可以不同意"克制才是真正的设计功力"（Layer 1），但仍然可以使用"按钮反馈 scale(0.97)"（Layer 4）。但体系的力量来自五层的对齐，当哲学、原则、规则、参数和文件位置全部指向同一个方向时，执行的确定性是单层规则无法达到的。

  • 待确认：Emil 在 Vercel 和 Linear 的具体工作经历如何影响了这套规则体系的形成，目前只能从技能内容中推断，缺乏直接的自述材料。

  • 反模式警示：将精确数值（如 cubic-bezier(0.23, 1, 0.32, 1)）当作放之四海皆准的银弹。Emil 自己在体系中也强调"cohesion matters"，数值必须符合产品的整体性格。机械地套用参数而不理解其背后的物理直觉，会导致正确但不协调的结果。
  • 反模式警示：将"删除动画"的立场极端化。体系明确区分了高频 UI（零动画）和罕见时刻（可以 delight），在信息传达关键的场景中（如状态变化指示），完全删除动画可能降低可用性。
