Waza：把工程习惯做成可执行技能

Waza 把八个工程习惯写成有边界、有证据、有停止条件的工作流。

这份报告基于本地 Waza 3.31.2 快照，分析八项技能如何覆盖工程生命周期，以及路由、规则、脚本、生成器和测试如何把个人经验整理成可分发、可校验的工程方法。

核心判断Waza 的克制体现在分工上：模型处理需要语境的判断，脚本检查可以明确判定的事实，验证器负责版本、路由、引用和分发一致性。八项技能各守一个边界，共同覆盖工程工作的关键时机。

关键发现

01八项技能按工程时机组织，不按工具分类
02判断留给技能，确定性检查交给脚本与规则
03结果、证据、完成条件和停止条件共同约束代理行为
04当前压力集中在入口密度和语义评估
01 / 定位

Waza 按工程时机组织技能

README 的定义很准：把工程师已经熟悉的习惯，变成 AI 代理可以执行的技能。仓库也按这条思路组织。

Waza 不按语言、框架或 IDE 分类，而是按工程时机排开。动手前用 think，界面工作用 ui，出现异常时用 hunt，交付前用 check。外部材料由 read 进入，经 learn 消化，再由 write 收束。health 负责检查代理配置、验证面和 AI 可维护性。

本地快照共有 202 个文件，包括 8 个 SKILL.md、26 份参考资料、25 个技能脚本、5 个专门审查或检查代理，以及 25 个 shell 测试。技能入口共 1,776 行，参考资料共 2,472 行，测试共 2,330 行。Waza 在技能内容和验证能力上投入了相近的篇幅。

快照事实

版本：3.31.2。本次运行 python3 scripts/verify_skills.py，全部检查通过。

本地目录不含 .git，因此本报告不推断提交历史、作者迭代顺序或当前远端版本。

02 / 八项技能

技能之间靠边界协作

八项技能各守一个决策边界，用户决定何时串联下一步。

技能处理的问题停止条件关键机制
think实施前完成价值、方案与交接判断方案获批后停止，不擅自实施Kill / Keep / Pivot、前提坍塌检查、阶段独立可合并
ui确立界面方向，或修复具体视觉缺陷必须检查真实渲染面，不能只凭代码声称完成把“丑、怪、突兀”当作审美校准，小修与方向调整分流
hunt先解释错误原因，再决定是否修复不能用一句话说明根因前，不改代码假设淘汰、精确复现、运行时证据阶梯、旧缓存边界
check审查 diff、项目质量、发布与远端收尾没有本轮验证证据，不得声称可合并或已发布深度分级、Safety Sink、发布矩阵、自动修复权限分级
read读取 URL、平台页面与 PDF抓取失败、遇到付费墙或空内容时必须说明按来源选择抓取器，把远端内容视为不可信数据
learn把陌生材料整理成可用的心智模型大纲未稳不能成文，发表动作需另行授权收集、消化、大纲、填充、精修、自审六阶段
write保留作者意图，去掉 AI 腔和本地化失真默认只交付成稿，不附自我解释按语言与文体加载参考，标点交给脚本检查
health审计代理配置、上下文、验证面与维护性先做预算友好的摘要检查，再决定是否深挖按项目复杂度校准要求，区分配置风险与代码腐化

技能默认不自动串联。think → 实施 → check、hunt → 修复 → check、read → learn → write 都由用户触发下一段。研究完成不会自动发布，方案获批也不会直接改代码。

排除边界

每个 frontmatter 都包含 “Not for …”。路由除了匹配触发词，还会明确排除不属于该技能的请求。

check 的入口最长：400 行、5 份参考、3 个脚本、2 个专门 reviewer。它承担的交付责任最多，上下文也最密。

03 / 方法

四个设计动作让技能可执行

Waza 把工程经验写进了代理的决策过程。四个设计动作解释了这套分工逻辑。

  1. 1
    先写完成标准，再动手

    每个技能开头就把结果、完成条件、证据和产出写清楚，把"做完了"变成可观察的事实，不让模型用一段叙述糊弄过去。

  2. 2
    用模式和停止条件守住边界

    think 区分轻量方案、价值判断和需求分诊，check 区分审查、项目体检、发布价值和发布执行。不同模式复用同一套立场，也明确什么时候必须停下，把决定交还给用户。

  3. 3
    把踩过的坑写成规则，事实判断留给当前证据

    Gotchas 把真实失误压缩成规则，下次同类场景自动触发。Durable Context 只保存偏好、原则和可迁移模式，代码、日志、截图、测试、CI 和远端状态仍需每次重新核验。

  4. 4
    能写脚本判定的，不留在自然语言里

    标点、引用、版本同步、路由漂移、包内容和生成物一致性都交给脚本。技能保留判断，脚本负责给出明确的通过或失败。

上下文预算

参考资料按需加载。最大的一组是 write：7 份参考、1,139 行；入口只负责选择该读哪一份。

目录型知识不能只增不减。近义规则需要合并，避免每次事故都变成一个新条目。

Waza 同时服务 Claude Code、Codex、Claude Desktop、Pi 和通用 skills 安装路径。多宿主支持最难的是保持一致。

测试密度

05 / 优势与代价

Waza 的压力开始落到注意力和评估上

当前快照中能看到几个压力点：技能持续积累后的上下文密度，以及缺少稳定的语义评估基线。

四个长期压力

  • 入口密度正在上升，check 已达 400 行，write 虽只有 228 行，却连接 1,139 行参考资料。规则越多，模型越可能抓住局部条款而丢失主任务。
  • 语义验证仍然偏弱，现有测试能证明版本、引用、脚本、包和若干行为烟测正确，但仓库未展示固定任务语料与跨模型结果评估。技能能否改善判断，仍主要依赖真实使用反馈。
  • 路由仍依赖自然语言，验证器确认触发词重叠低于阈值，却无法消除“看看这个”“判断一下”这类依赖上下文的歧义。
  • 经验可能过拟合，仓库明确要求移除私有路径和项目事实，这能控制泄漏，却不能证明少数项目提炼出的规则适用于更广环境。
判断边界

这些问题描述的是长期维护压力，不影响本次结构与分发校验结果。

“八项技能硬上限”是有效治理，但并不自动阻止单个技能继续膨胀。

06 / 学习路径

先看边界，再读具体条款

想把 Waza 变成自己的工作方式，可以从 Resolver 开始，再进入主干技能、规则和验证器。

第一轮从 Resolver 与 README 建立地图

重点看歧义消解和手动串联，先弄清一个请求为什么属于某项技能。

第二轮精读 think、hunt、check

这三项构成动手前、出问题和交付前的工程主干，可以对照它们的证据要求与停止条件。

第三轮读 rules 与 verifier

观察哪些经验进入全局规则，哪些事实被写成确定性检查。这一轮最接近 Waza 的设计思想。

第四轮选择最贴近当前工作的技能

从 UI、Write、Read、Learn、Health 中选一项，用当前任务跑一遍，更容易看清技能带来的差异。

第五轮做一次启用前后对照

让同一任务分别在启用与不启用技能时执行，比较遗漏、追问、证据、越权和返工，把差异记录成行为。

复制一项能力，可以沿用这个骨架

Outcome    最终要改变什么
Done       什么证据表明已经完成
Boundary   什么不属于这个技能
Modes      哪些场景需要不同流程
Hard Rules 哪些约束必须改变默认行为
Gotchas    哪些真实失败值得写成规则
Scripts    哪些事实可以交给脚本判定
Stop       何时必须交还用户决定
阅读起点

skills/RESOLVER.md 只有 92 行，却解释了八项技能的相互边界，适合作为第一份材料。

先记录自己的重复失败，再决定哪些经验值得写成规则，不必照搬 Tw93 的具体 Gotchas。

前提假设

  • 分析对象是本地 Waza 3.31.2 快照。
  • 仓库文件反映作者在该快照中的公开设计意图。
  • 结构与脚本可以证明工程化程度，无法单独证明跨模型效果。

待确认

  • 远端仓库是否已有晚于 3.31.2 的变化。
  • 八项技能在不同模型与宿主上的稳定收益。
  • check 与 write 是否已接近需要继续下沉参考资料的阈值。

建议处理

  • 学习顺序可以从 Resolver、主干技能、Rules 走到 Verifier。
  • 用真实任务做启用前后对照，记录行为差异。
  • 只沉淀反复出现的失败，不照搬其他项目的命令和边界。
