Hermes Agent 架构分析报告

一个自学习 AI Agent 的架构报告

Hermes Agent 是 Nous Research 构建的自改进 AI agent。它横跨 24+ 消息平台、40+ 工具、30+ 模型供应商和 6 种终端后端，核心循环却做得很克制。本报告分析其架构决策、学习闭环和安全姿态。

核心判断 Hermes Agent 有意思的地方不在功能多，在于它同时做了两件通常互斥的事：边缘层拼命扩张（平台、工具、供应商），核心层却收得很紧（极简核心、缓存保存、精确依赖锁定）。这种扩张与克制并存的结构，是它区别于其他 agent 框架的地方。

架构总览

核心循环：极简核心，能力在边缘

Hermes 的核心 agent 循环简单且克制：收消息 → 构建系统提示 → 调 LLM → 分发工具调用 → 返回结果。它真正的分量不在做了什么，在它拒绝往这个循环里塞东西。

项目不小：cli.py 738KB，run_agent.py 268KB，hermes_state.py 253KB，gateway/run.py 967KB。但核心 agent 循环被严格限制在约 5,000 行的 agent/conversation_loop.py 中。AGENTS.md 的说法很直接：

The core is a narrow waist; capability lives at the edges. Every model tool we add is sent on every API call, so the bar for a new core tool is high.

简单说就是核心保持极简，能力放在边缘层。每往核心加一个工具，它就会出现在每一次 API 调用中，吃 token、还可能打断 prompt 缓存。所以 Hermes 建了一套严格的"足迹阶梯"（Footprint Ladder），新增能力的优先级从高到低：

  1. 扩展现有代码（不增加新工具 schema）
  2. CLI 命令 + Skill（工具逻辑在 skill 脚本里，模型看不到 tool schema）
  3. 按条件启用的工具（check_fn 控制可见性，只在条件满足时才出现在 tools 列表中）
  4. Plugin（独立注册，可通过配置禁用）
  5. MCP 服务器（外部进程，按需连接）
  6. 新建核心工具（最后手段）

Agent 循环的四个阶段

Turn Prologue，构建或恢复系统提示（首次构建后缓存，绝不在会话中途重建，以保证上游 prefix cache 持续命中）；预检上下文是否需要压缩；注入记忆快照和用户画像；执行 pre_llm_call 插件钩子。

API 调用，对消息列表做深度清洗（修复 JSON、修复角色交替违规、剥离孤立工具结果、合并相邻用户消息），然后通过传输适配器发起流式调用。支持四种 API 模式：chat_completions、anthropic_messages、bedrock_converse、codex_responses，每种有独立的响应验证和错误恢复路径。

工具执行，验证工具调用名称和 JSON 参数（自动修复常见拼写错误和截断检测，最多重试 3 次），决定串行或并发执行（检查工具间依赖关系），在 ThreadPoolExecutor（最多 8 个工作线程，420 秒超时）中并发调度。

Turn Finalization，持久化消息到 SQLite SessionDB（WAL 模式，FTS5 全文搜索），更新 token 计数和成本估算，执行 post_turn 插件钩子。

代码证据

核心循环定义在 agent/conversation_loop.py:518 的 run_conversation() 函数中，约 4,600 行的单一函数。尽管长度惊人，但其结构清晰：prologue → loop（API call → tool dispatch）→ finalize。

这些设计约束有实际的经济账：每次 prompt 缓存未命中都会成倍增加 API 费用，对长对话来说，缓存命中和未命中的成本差距可达 10 倍以上。

学习闭环

记忆、技能与会话搜索的三层架构

Hermes 声称自己是"唯一具有内置学习循环的 agent"。这并非营销语言，它确实构建了一个从经验中提取、固化、检索知识的三层系统。

第一层：持久记忆（Memory）

memory_tool.py 提供两个独立的知识库：MEMORY.md（agent 的私人笔记）和 USER.md（用户画像）。关键设计决策是冻结快照模式：会话启动时拍下记忆快照注入系统提示，会话中途的写入立即落盘但不更新系统提示，这保护了 prompt 缓存，代价是当前会话看不到本会话内的记忆更新。合并操作通过 budget 感知的批处理 API 完成，超限时自动拒绝以防止模型在合并-重试循环中耗尽 token。

第二层：程序性记忆（Skills）

Skills 系统是 Hermes 学习机制里最有想法的部分。它包含三个独立的生命周期阶段：

创建，agent 在完成复杂任务后自主创建 skill（通过 skill_manage 工具的 create 动作），将成功路径固化为可复用的 SKILL.md 文件（含 YAML frontmatter，符合 agentskills.io 开放标准）。

改进，skill 在使用中通过 patch 动作（基于 fuzzy_match.py 的模糊查找替换）自我改进。每次写入经过 skills_guard.py 安全扫描，恶意内容触发回滚。

策展，一个自主"策展人"（autonomous curator）后台进程读取使用遥测数据（skill_usage.py），决定 skill 的生命周期转换：活跃 → 陈旧 → 归档/固定。策展人只能操作自己创建的 skill，不能触碰外部安装或 hub 来源的 skill。

第三层：会话搜索（Session Search）

session_search_tool.py 基于 SQLite FTS5 提供跨会话的全文搜索，让 agent 能够"回忆"过去的对话。搜索结果经过 LLM 摘要后再注入当前上下文，避免原始对话历史撑爆上下文窗口。

Honcho 用户建模

Hermes 集成了 Honcho 的辩证用户建模，一种通过对话历史推导用户信念、偏好和思维模式的系统。这为记忆系统提供了一个不断深化的用户画像。

关键数据

Skills Hub 接入 9 个来源适配器，搜索范围覆盖 50,000+ 社区 skill（ClawHub）、20,000+ skill（skills.sh）、14,500+ agent 模板（LobeHub）。所有外部安装经过隔离扫描流程：下载 → 隔离区 → 安全扫描 → 安装或拒绝。

冻结快照模式是"用限制换收益"的典型：牺牲了会话内的记忆即时可见性，换来 prompt 缓存持续命中。对长对话来说，这个取舍是划算的。

平台覆盖

Gateway：消息平台的三层集成架构

Hermes 支持 24+ 消息平台，从 Telegram 到企业微信，从 Signal 到飞书。这种广度不是通过堆砌适配器实现的，它建立在一套精心设计的三层集成架构上。

内置适配器（Built-in）

核心团队维护的平台适配器，位于 gateway/platforms/。包括 Telegram（71KB）、Discord、Signal（71KB）、WhatsApp（Baileys 桥 + Meta Cloud API 双通道）、企业微信（92KB）、元宝（216KB）、QQBot（124KB）、飞书等。每个适配器实现 BasePlatformAdapter 抽象基类，这是一个 245KB 的庞然大物，定义了完整的适配器契约：连接/断开、发送消息/图片/文件/语音、输入状态指示、交互式 UX（审批、模型选择）、流式草稿更新等。

Plugin 适配器（推荐社区路径）

社区贡献者只需在 plugins/platforms/<name>/ 下创建目录，包含 plugin.yaml 清单和适配器实现，不需要修改任何核心代码。Plugin 系统通过文件系统扫描自动发现，支持延迟加载（重 SDK 如 discord.py 只在首次使用时导入）。当前已有 17 个社区平台 plugin：钉钉、邮件、Google Chat、IRC、LINE、Matrix、Mattermost、ntfy、Photon、Raft、Simplex、Teams 等。

Relay 适配器（实验性）

一个通用的 RelayAdapter 通过 WebSocket 连接远程"连接器"进程。连接器在握手时发送 CapabilityDescriptor 声明自己是哪个平台，Relay 适配器据此呈现对应的平台行为。这意味着 Hermes 本身不需要任何平台 SDK。

Gateway 运行器

gateway/run.py（967KB）是 gateway 进程的大脑：管理适配器生命周期、维护 AIAgent 实例的 LRU 缓存（最多 128 个，1 小时空闲 TTL）、处理会话自动恢复（中断后 1 小时新鲜窗口内自动续接）、桥接同步 agent 回调和异步平台投递（通过 queue.Queue + asyncio 任务）。流式投递支持三种传输模式：edit（渐进式编辑消息）、draft（原生草稿 API）、auto（优先草稿回退编辑）。所有外发消息经过 Tirith 级敏感文本编辑 + 正则兜底扫描。

平台枚举

gateway/config.py 中的 Platform 枚举通过 __missing__() 支持动态成员，任何 plugin 注册的平台名称自动成为有效的枚举值，无需修改枚举定义。

流式事件词汇

gateway/stream_events.py 定义了类型化的事件词汇（MessageChunk、ToolCallChunk、ToolCallFinished、LongToolHint），将 agent 核心与平台特定投递解耦。

工具系统

工具注册、分批暴露与安全护栏

93 个工具文件，但核心循环只看到一个动态装配的工具数组。Hermes 的工具系统在"丰富"和"克制"之间拿捏得很准。

自注册 + 语法树扫描

每个工具文件在模块导入时调用 registry.register() 自行注册。discover_builtin_tools() 用语法树扫描（AST parse，而不是直接 import）来发现哪些模块含注册调用，然后才导入它们。这避免了无条件导入所有工具模块的启动开销。

状态检测 + 短期缓存

每个工具可附带一个 check_fn 状态检测函数，探测外部依赖是否就绪（Docker daemon、Modal SDK、Playwright 二进制）。检测结果缓存约 30 秒，失败后给 60 秒宽限期，防止瞬时故障（比如 Docker 偶发超时）在会话中途把工具悄悄摘掉。

分批暴露（Tool Search）

当 MCP 和 plugin 工具数量超过阈值（默认上下文窗口的 10%），它们会被替换为三个桥接工具：

  • tool_search(query, limit)，对延迟加载的工具目录做 BM25 文本检索
  • tool_describe(name)，返回单个工具的完整 JSON schema
  • tool_call(name, arguments)，调用延迟工具（路由与直接调用完全相同）

核心工具永远不会被延迟。CLI 活动面板、gateway 和保存的轨迹自动解包桥接工具调用为底层工具名称，对用户透明。

安全护栏

安全系统贯穿多个维度：

  • 命令审批（approval.py，129KB）：危险命令在执行前需要用户审批，支持正则模式匹配的 allowlist
  • 写入审批（write_approval.py）：记忆和 skill 写入可以路由到 pending/ 目录等待审核
  • 路径安全（path_security.py）：防止路径遍历攻击
  • URL 安全（url_safety.py）：SSRF 防护
  • 威胁扫描（threat_patterns.py）：共享威胁模式定义，用于记忆内容和 skill 内容扫描
  • Tirith 集成（tirith_security.py）：策略引擎集成，用于外发消息的敏感文本编辑
  • Skill 守卫（skills_guard.py，44KB）：外部 skill 安装前的安全扫描
供应链防御

pyproject.toml 中的所有直接依赖使用精确锁定（==X.Y.Z），这是对 2026 年 5 月 Mini Shai-Hulud 蠕虫事件的直接响应。Provider 特定的包被移到 extras 中并通过 lazy_deps.py 延迟加载，缩小了供应链攻击面。

精确依赖锁定本身不算新鲜。值得注意的地方是他们把这次安全事件的教训写进了代码注释和架构约束里，从单次事件提取出了可执行规则。

部署矩阵

六种终端后端与无服务器持久化

Hermes 可以在 $5 VPS、GPU 集群或无服务器基础设施上运行，空闲时成本几乎为零。这不是偶然，是六种独立终端后端主动设计出来的结果。

后端隔离级别持久化适用场景
Local无（共享主机）本地文件系统日常使用、开发
Docker容器级Volume 挂载生产部署、s6-overlay 监督
SSH远程主机级远程文件系统在远程服务器上运行
Singularity容器级（HPC）绑定挂载学术 HPC 集群
Modal无服务器容器卷（hibernates）按需唤醒，空闲零成本
Daytona无服务器容器卷（hibernates）开发环境即服务

Docker 部署：s6-overlay 监督

官方 Docker 镜像基于 Debian 13.4，使用 s6-overlay 作为 PID 1 进行进程监督。关键安全设计：/opt/hermes 挂载为 root 所有的不可写目录。非 root 用户（UID 10000，可通过 HERMES_UID 覆盖）运行 agent 进程。启动时自动检测 Docker socket 组成员身份以支持 docker-in-docker。

Nix 部署：从 flake 到 NixOS module

nix/nixosModules.nix（45KB）提供了声明式的 NixOS 部署：

  • Native systemd 模式：services.hermes-agent.enable = true，通过 settings attrset 声明式配置，支持 mcpServers 声明、hostUsers 多用户设置
  • Container 模式：在非 NixOS 主机上，将 /nix/store 绑定挂载到 Ubuntu 容器中，首次启动时自动配置 nodejs、uv、Python 3.12 venv
版本标识

当前版本 0.18.0（2026.7.1）。项目仍处于快速迭代期，但核心架构已经稳定。有 AGENTS.md 这件事本身就说明架构已经趋于成熟。

前提假设

  • 分析基于 2026 年 7 月 2 日的 main 分支快照，未运行测试套件或实际部署
  • 架构判断来自代码阅读和 AGENTS.md 中的设计意图声明，非来自性能测量
  • 极简核心设计是否在长对话中确实保持了 prompt 缓存命中率，需要生产环境的 API 日志验证
  • Skills Hub 的 9 个来源适配器中，部分（skills.sh、ClawHub）的可用性取决于第三方服务的持续运营

待确认

  • gateway/run.py（967KB）和 cli.py（738KB）是否需要进一步模块化，它们是目前最大的单文件
  • Tool Search 的 BM25 检索在工具数量达到数百个时的召回质量，缺乏公开基准
  • Honcho 用户建模的实际准确性和对 prompt 缓存的性能影响
  • Relay 适配器仍标记为实验性，其安全模型（远程连接器可声明任意平台身份）在生产环境中是否充分

建议关注

  • 监控 agent/conversation_loop.py 的渐进提取进度，目前约 4,600 行的单一函数仍有进一步拆分的空间
  • 关注 Tool Search 的 BM25→语义检索升级路径，当工具目录增长时这可能成为瓶颈
  • 评估 Memory 冻结快照模式是否适合需要"会话内即时记忆更新"的使用场景
  • 如果考虑自行部署，优先评估 Nix flake 路径，它的声明式配置和可重复构建是运维优势
