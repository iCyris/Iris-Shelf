Claude Code 的四种循环

从手工检查到全自动代理：四种循环如何重新定义人与 AI 的分工边界

Claude Code 团队将"循环"定义为智能体重复执行工作直到满足停止条件的模式。它是一套渐进式的自主权委托体系。每一步，你交出去的东西都不同。

核心判断 选对循环类型比写好 prompt 更重要。四种循环覆盖了"人在回路"到"人退出回路"的完整谱系，关键是理解每种模式下你把什么决策权交给了 Claude，以及你保留了哪些验证手段。

关键发现

01 四种循环各有专属分工：基于轮次、目标、时间和主动式，从手工到全自动逐级递进
02 每升一级，"交出去"的东西不同：检查权→停止条件→触发→整条 prompt
03 循环质量取决于周边系统：代码库清洁度、可验证技能、文档可达性、系统化改进
04 Token 消耗可控：选对原语、设定明确停止条件、小规模试跑、匹配变更频率
01 / 概念

什么是循环？

循环就是 Claude 重复执行"感知→行动→检查"的工作周期。每一次 prompt 提交，都是一次循环的启动。

Claude Code 团队将循环按四个维度分类：触发方式、停止条件、底层原语和适用任务类型。这四个维度串起来，就是从"人在回路中"到"人完全退出回路"的完整谱系。

大部分开发者从基于轮次的循环开始，也就是日常的 prompt 对话。工作流真正发生变化是在后面三种循环里：你不再亲自检查结果、不再亲自决定何时停止、甚至不再亲自触发任务。到这个阶段，Claude 不再只是一个工具。

循环类型 你交出去什么 适用场景 可用原语
基于轮次 检查权 探索、决策中 自定义验证技能
基于目标 停止条件 你知道"完成"长什么样 /goal
基于时间 触发 定期发生的外部事件 /loop、/schedule
主动式 整条 prompt 可重复定义的工作流 以上全部 + 动态工作流
来源

Delba de Oliveira & Michael Segner, "Getting Started with Loops", Claude Blog, 2026年6月30日。文章归属于 Claude Code 分类。

四个维度的分类框架来自原文，这不是事后归纳，而是团队设计这些原语时的意图。理解这个框架比记住命令更重要。

02 / 逐级递进

四种循环，四种委托契约

基于轮次：最熟悉的循环

每一次 prompt 就是一次循环。Claude 读取上下文、执行操作、检查结果、需要时重复、然后回复。你手动检查并写下一条 prompt。这是人机协作的默认模式。

这种模式的问题是：Claude 对"完成"的判断取决于自己的主观评估。如果它认为改好了但实际没改好，你不会发现，除非你亲自验证。原文的建议是用 SKILL.md 把验证步骤编码为可重复的检查流程，让 Claude 自己跑一遍浏览器、看一遍控制台、跑一遍性能审计，而不是基于"编辑成功"就说完成。

# 验证前端变更
绝不只因编辑成功就报告 UI 变更完成。按以下步骤验证：

1. 启动 dev server 并在浏览器中打开编辑过的页面
2. 直接与新控件交互：点击、确认状态变化、截图前后对比
3. 检查浏览器控制台：零新增错误或警告
4. 使用 Chrome DevTools MCP 运行性能跟踪并审计 Core Web Vitals

若任一步失败，修复后从第 1 步重新执行。

基于目标：交出"停止"的判断权

当你用 /goal 定义了成功标准，Claude 不再自行判断什么叫"够好"。每次它想停下来时，一个评估模型会检查你的条件，若未达标就把它送回工作岗位，直到目标达成或达到最大轮次。

原文的态度很明确：确定性的标准最有效，比如测试数量、分数阈值，具体的可度量指标。模糊的目标（"让代码更好"）在这套机制下效果不佳。

/goal 把首页 Lighthouse 分数提高到 90 分以上，最多 5 次尝试。

基于时间：交出"触发"的判断权

/loop 按固定时间间隔触发任务。原文列举了两类典型场景：周期性重复工作（每天早晨总结 Slack 消息）和与外部系统交互（每隔几分钟检查 PR 是否收到新评论或 CI 是否失败）。

/loop 5m 检查我的 PR，处理 review 意见，修复失败的 CI

需要注意：/loop 在你的电脑上运行，关了就停了。/schedule（研究预览阶段）可以将循环移至云端，成为真正的无人值守任务。

主动式：交出一切

这是四种原语的组合体：/schedule 定时检查、/goal 定义完成标准、动态工作流并行探索方案、自动模式消除权限询问。适用于定义清晰、重复出现的工作流：bug 分类、issue 分流、依赖升级。

/schedule 每小时：检查 #project-feedback 中的 bug 报告。 /goal：本轮找到的每个报告都要完成分类、修复和回复，否则不停止。修复 bug 时，使用工作流在并行工作树中探索三个方案，并让评判代理做对抗性审查。

委托谱系

轮次循环：检查→目标循环：停止条件→时间循环：触发→主动循环：全部。每升一级，人的介入减少但需要更精确的验证机制。

确定性标准是这套体系里最重要的设计约束。越往自动化走，"完成"的定义越不能靠 Claude 自己说了算。

03 / 质量工程

代码质量不是循环能自动保证的

循环的输出质量取决于它周围的环境。原文给了五条实践：

  1. 1
    保持代码库整洁

    Claude 遵循已有的模式和约定。混乱的代码库产生混乱的循环输出。Claude 的编辑策略直接受上下文代码风格影响，没有例外。

  2. 2
    给 Claude 自验证的能力

    用 Skill 编码质量标准，让 Claude 能够"看见、测量或与结果交互"。量化的检查比主观判断可靠，可重复的检查比一次性验证可靠。

  3. 3
    让文档触手可及

    框架和库的文档应包含最新的最佳实践。这不是文档工程师的事，Claude 读这些文档来决定怎么实现。

  4. 4
    用第二个代理做代码审查

    新鲜上下文的审查者偏见更少，不受主导代理推理路径的影响。使用内置的 /code-review 技能或 Code Review for GitHub。

  5. 5
    把个案改进编码为系统修复

    当一个单独结果不达标准时，修复系统而不是只修这个结果。把教训编码到 Skill 或 CLAUDE.md 中，让所有后续循环受益。这是循环里最容易被忽略的一步。

第 5 条是原文最关键的工程洞察。单次修复修掉一个 bug，系统性改进修掉一类 bug。放到循环里看，系统性改进的收益会被每次迭代放大。

04 / Token 经济

六条 Token 管理策略

循环会消耗更多 token，这是自主性的代价。管好了，这个成本可控也可预测。

策略 做法 为什么有效
选对原语和模型 小任务不需要多代理或多轮循环；部分任务可用更快更便宜的模型 不是所有任务都需要最强模型或最复杂编排
定义清晰的成败标准 精确描述"完成"的样子，让 Claude 更快抵达方案且不过早停止 模糊目标导致反复试错和过早退出
先试跑再铺量 动态工作流可能生成数百个代理，先在小子集上评估消耗 小样本的 token 消耗乘以 N，就不需要靠猜
用脚本替代推理 运行已有脚本比重新推导便宜，例如 PDF 技能可以打包表单填充脚本 推理消耗远大于执行消耗
匹配变更频率 不要比需要更频繁地运行例行任务 无变更的空跑是纯浪费
使用用量审计工具 /usage 按技能、子代理、MCP 细分；/goal 无参数显示轮次和 token；/workflows 逐代理显示消耗 可观测才有可管理

最反直觉的一条是"用脚本替代推理"。开发者常觉得让 AI 推导代码才是物尽其用，但原文的判断是：已有确定方案的步骤，直接执行比重新推导便宜得多。这块是循环成本控制里容易被忽略的大头。

成本与自主性的权衡

每提升一级循环的自主性，token 消耗都会增加。这是设计意图，不是缺陷。关键是让增加的消耗对应真实的价值，不是用最高级循环做一个简单的文本替换。

审计命令

/usage 近期用量按 skill、子代理、MCP 细分
/goal 当前轮次和 token 数
/workflows 逐代理 token 消耗

你在探索或决策中 基于轮次

配合验证 Skill 提高轮次效率

你知道完成长什么样 基于目标

需要确定性、可度量的停止条件

工作按时间表发生 基于时间

固定间隔检查外部变化并响应

工作是重复出现且定义清晰的 主动式

组合所有原语，用工作流和自动模式

05 / 落地路径

从哪个任务开始？

原文的建议是：不要追求一次性全面自动化。找一个你当前是瓶颈的任务，问三个问题。

  • 你能写出验证检查吗？如果你不能清晰定义"成功"，就不要用目标循环。
  • 目标够清晰吗？如果"完成"长什么样都说不清，轮次循环更适合。
  • 工作是按时间表来的吗？如果是定期发生的，就该考虑时间循环。

这三个问题本身就是决策树。答案决定你从哪个层级开始，往上走到哪一级。原文最后一句值得记住："运行循环，观察它在哪卡住或过度执行，不要害怕迭代它。"循环本身也是可迭代的，它也是代码。

原文的"不要害怕迭代"是写给所有准备尝试循环的开发者。循环不是一次性配置，它需要像代码一样维护、观察和改进。

前提假设

  • 假设读者已具备基本的 Claude Code 使用经验，理解 prompt、skill、子代理等基础概念
  • 条款默认开发者有可运行的 CI 和验证流程，循环的质量依赖这些基础设施
  • 原文写作时间点的功能描述（如 /schedule 处于 research preview 阶段）可能在阅读时已发生变化
  • Token 消耗数据基于 Claude Code 当前定价模型，实际成本需结合个人用量确认

待确认与边界

  • /schedule 仍在研究预览阶段：云端无人值守循环的可靠性和成本尚未有足够多的生产案例
  • 主动式循环对团队协作的冲击未充分讨论：当多个人的循环同时操作同一个代码库时，冲突如何解决
  • 评估模型的判断标准与人类判断的差异尚未量化，"达标"的机器定义可能与你的直觉存在差距
  • 循环体系对安全边界的影响（自动模式跳过权限确认）需要额外的权限策略考量

建议处理

  • 选择一个你当前手工处理的重复任务，用三个问题（验证检查、目标清晰度、时间触发）判断它适合哪种循环
  • 先为现有轮次循环编写一个验证 Skill，这是投入最低、回报最确定的改进
  • 在升级到 /goal 循环之前，确认你有一个可量化、可自动验证的成功指标
  • 使用 /usage 建立基准，在升级循环层级后对比 token 消耗，确认增加的自主性对得起增加的成本
